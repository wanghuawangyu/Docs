## 图例

- **真彩色**：支持 `\e[38;2;R;G;Bm` / `\e[48;2;R;G;Bm` SGR 序列以实现 1600 万色
- **256 色**：支持 `\e[38;5;Nm` / `\e[48;5;Nm` 索引色
- **Unicode/Emoji**：完整 = 正确的字形聚类、正确的双宽、emoji ZWJ 序列；部分 = 基础多字节但宽度错误或缺失序列
- **Kitty 键盘协议**：支持 `CSI > u` 渐进增强键盘协议
- **带括号粘贴**：支持 `\e[?2004h` 以在粘贴内容前后包裹开始/结束标记
- **鼠标捕获**：支持 SGR 1006 鼠标报告（`\e[?1006h`）
- **备用屏幕**：支持 `\e[?1049h` 备用屏幕缓冲区

## 已知导致白色块或陈旧内容的渲染问题

### 1. 背景色溢出 / “白色块”

**根本原因**：当 TUI 在单元格上设置了背景色，但终端在下一帧未能正确清除或重新绘制该单元格时，该单元格会保留其旧内容或回退到默认背景（浅色主题下通常为白色）。

**受影响的终端和场景**：
- **所有终端**：如果应用写入 `\e[K`（清除至行尾）而未先通过 SGR 设置正确的背景色，则被清除区域继承终端的默认背景，而非应用期望的颜色。这是暗色主题 TUI 中出现白色/浅色块的首要原因。
- **tmux**：tmux 仿真自身的屏幕缓冲区。如果内部应用使用 BCE（背景色擦除），而 tmux 的默认终端未正确通告 BCE 支持，则被擦除区域会渲染为错误的背景色。**修复**：确保使用 `tmux-256color` terminfo，并使其与外层终端的能力匹配。
- **Windows Terminal (ConPTY)**：ConPTY 有时会错误地合并快速的 SGR+擦除序列，导致调整大小或快速重绘后，行边界处的 1-2 个单元格保留旧的背景色。
- **VS Code Terminal (xterm.js)**：基于 Canvas 的渲染器在终端在普通屏幕和备用屏幕缓冲区之间快速交替时，可能留下“幽灵”单元格，尤其在调整大小事件期间。

### 2. Emoji / 双宽字符错位

**根本原因**：终端和应用对字符占用的列数判断不一致。应用认为 emoji 占 2 列宽（基于 Unicode East_Asian_Width），但终端将其渲染为 1 列（或反之），导致该行后续每个单元格都发生偏移。

**受影响的终端**：
- **macOS Terminal.app**：尤其严重。许多 emoji 渲染为 1 列宽，而应用（使用 libc wcwidth 或 Unicode 表）假定为 2 列。这使整行失步，留下显示为空白/白色块的“幽灵”单元格。
- **tmux**：tmux 有自己的内部 wcwidth 实现。如果它对外层终端关于字符宽度的判断不一致（在较新 Unicode 版本中添加的新 emoji 上很常见），光标定位会出错，单元格会显示为重复或空白。
- **screen**：比 tmux 更糟。其 Unicode 宽度表已过时多年。大多数 emoji 和许多 CJK 字符会破坏行布局。
- **Alacritty**：总体良好，但属于 PUA（私用区）码位的 Nerd Font 字形默认为 1 列宽。如果应用假定为 2 列，则会发生错位。

### 3. 调整大小后的陈旧内容

**根本原因**：当终端窗口调整大小时，应用收到 `SIGWINCH` 并必须重绘。如果重绘是部分的，或者终端的行重排逻辑与应用的假设冲突，旧内容仍然可见。

**受影响的终端**：
- **Konsole**：调整大小时的重新流动很激进——它会重新流动软换行的行，这可能与期望每行独立的 TUI 应用冲突。这会导致瞬时的“双重渲染”伪影。
- **tmux**：调整大小导致 tmux 重新流动自身缓冲区，然后向内部应用转发 `SIGWINCH`。存在竞争条件：如果应用在 tmux 完成重新流动之前重绘，旧内容会显示 1-2 帧。
- **VS Code Terminal**：xterm.js 的大小调整处理程序可能落后于实际的视口大小，导致应用在 1-2 帧内按错误的尺寸绘制。

### 4. 备用屏幕缓冲区切换伪影

**根本原因**：在进入或离开备用屏幕（`\e[?1049h` / `\e[?1049l`）时，某些终端未完全清除缓冲区，或恢复了错误保存的状态。

**受影响的场景**：
- **Warp**：Warp 的基于块的架构不使用传统的备用屏幕。依赖 `\e[?1049h` 的 TUI 应用可能会发现其输出与 Warp 的 shell 集成 UI 元素混合在一起。
- **tmux + 嵌套会话**：嵌套的 tmux 会话（或 tmux 内嵌 screen）可能丢失对当前活动备用屏幕缓冲区的跟踪，导致外层多路复用器的状态栏覆盖在内部应用的内容上。
- **macOS Terminal.app**：在旧版 macOS（Ventura 之前）上，从备用屏幕恢复时偶尔会使光标保持不可见，直到用户输入。

### 5. 光标可见性问题

**根本原因**：`\e[?25l`（隐藏光标）和 `\e[?25h`（显示光标）并不总是可靠配对，尤其在应用崩溃或被 SIGKILL 杀死时。

**受影响的终端**：
- **所有终端**：如果 TUI 应用崩溃而未恢复光标，光标会保持隐藏。大多数现代终端（kitty、WezTerm、iTerm2）在 shell 提示符下自动恢复，但 GNOME Terminal 和 Terminal.app 可能使光标保持隐藏，直到执行 `reset` 或 `tput cnorm`。
- **tmux**：如果内部窗格的应用隐藏了光标，然后用户切换窗格，光标可见性状态可能在窗格间泄漏（在较新 tmux 版本中已修复，但在 3.3 及更早版本中仍可观察到）。

### 6. SGR 重置范围问题

**根本原因**：`\e[0m`（SGR 重置）应重置所有属性，但某些终端在处理下划线样式、下划线颜色或删除线时表现不一致。
- **GNOME Terminal (VTE)**：旧版 VTE 在 SGR 0 时未重置下划线颜色，导致彩色下划线跨行持续。
- **Konsole**：历史 bug 中 `\e[0m` 未重置上划线属性。
- **screen**：`\e[0m` 不能可靠地重置 256 色前景/背景，导致后续文本残留旧颜色。

### 7. Kitty 键盘协议回退问题

**根本原因**：启用 Kitty 键盘协议但未在退出时（或崩溃时）正确禁用的应用，会使终端停留在增强键盘模式。随后的 shell 输入可能产生 garbled 转义序列。
- **Kitty、Alacritty、WezTerm、Ghostty**：如果应用未在退出时调用 `CSI < u`，均受影响。Kitty 本身在检测到 shell 提示符时自动重置。Alacritty 和 WezTerm **不**自动重置——用户必须运行 `reset`。

### 8. tmux 特定的透传限制

tmux 是 TUI 应用中最常见的渲染问题来源，因为它插入了一个完整的 VT100 仿真层：
- **转义序列过滤**：tmux 剥离任何它不认识的转义序列。这会破坏 Kitty 键盘协议、Kitty 图形协议、iTerm2 内联图像以及某些扩展 SGR 属性（例如，`CSI 4:3 m` 花式下划线需要 tmux 3.4+）。
- **延迟透传**：即使设置了 `set -g allow-passthrough on`，DCS 透传也会增加延迟，并可能分割长序列。
- **TERM 不匹配**：如果内部 `TERM` 与 tmux 通告的能力不匹配（例如，应用看到 `xterm-256color` 但 tmux 仅透传 `screen-256color`），颜色/能力协商会静默失败。
- **剪贴板**：OSC 52 剪贴板支持可用，但必须显式启用（`set -g set-clipboard on`）。

## 给 TUI 开发者的建议

- **在擦除前始终设置背景色**：在任何 `\e[K`、`\e[J` 或 `\e[2J` 之前，通过 SGR 设置预期的背景色。永远不要假设终端的默认背景与你的主题匹配。
- **使用 `COLORTERM` 检测真彩色**：检查 `COLORTERM=truecolor` 或 `COLORTERM=24bit`，而非解析 terminfo——后者对 RGB 支持不可靠。
- **防御性地处理 emoji 宽度**：使用 Unicode 15.1+ 宽度表，并接受某些终端会与之不一致。考虑在网格对齐的 TUI 布局中避免使用 emoji，或使用显式空格填充。
- **在 `SIGWINCH` 时完全重绘**：不要在调整大小时尝试增量修补屏幕。清除所有内容并从头重绘。
- **退出时始终恢复终端状态**：使用清理处理程序（甚至对 SIGTERM/SIGINT 也有效）来：恢复光标可见性、离开备用屏幕、禁用鼠标捕获、禁用带括号粘贴、重置 Kitty 键盘协议，并发出 SGR 重置。
- **在 tmux 下测试**：如果你的用户可能在 tmux 内运行，请在那里显式测试。许多渲染错误仅在多路复用器下出现。
- **对 Terminal.app 和 screen 优雅降级**：这些仍然是常用终端中能力最低的。通过 `TERM_PROGRAM` 或 `TERM` 检测它们，并回退到 256 色模式及 ASCII 安全的 UI 元素。
