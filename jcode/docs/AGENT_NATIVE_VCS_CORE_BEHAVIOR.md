# 代理原生 VCS：核心行为

**状态：** 设计草案，整合了本次会话的核心思想。

## 一句话概括

此系统本质上不是一种更好的合并算法。它是一个 Git/jj 层的 VCS，能够保留本地变更的语义、所有权和维护上下文，使智能代理能够持续相互协调，并在上游变动时重新维护本地增量。

## 核心框架

- Git 主要是分支优先的。
- jj 主要是变更优先的。
- 此系统应是 **车道优先** 和 **维护包优先** 的。
- 底层存储和传输可以保持与 Git 兼容。创新之处在于 VCS 使其成为一等公民的元数据、分组和工作流。

## 此 VCS 要解决的问题

有两个相关但不同的问题：

**并行多代理编辑同一代码库**
- 代理在短时间内进行连贯的编辑工作。
- 它们的工作可能交错并产生冲突。
- 匿名的脏状态会导致犹豫和污染。

**长期本地定制与上游变动**
- 用户将越来越多地维护开源项目的个性化下游变体。
- 困难不在于补丁应用，而在于在上游变动时保留本地增量的意图。

## 设计目标

- 使代理工作能够自然地表示。
- 消除匿名脏状态。
- 为未来的代理保留足够的上下文，以维护相对于上游的本地变更。
- 保持机器历史的丰富性，同时允许面向人类的历史保持整洁。
- 保持与 Git 生态系统和远程托管的兼容性。

## 非目标

- 不试图首先取代 Git 对象存储。
- 不承诺所有主要的上游分歧都能自动解决。
- 不依赖原始推理轨迹作为理解的主要工件。

## 核心实体

### 1. 车道（Lane）

车道是进行中工作的主要单元。车道通常由以下内容键控：
- 目标（goal）
- 代理（agent）

对于下游维护，车道可以是一个长期定制的车道，代表持久的本地增量。车道不仅仅是一个标签，它还具有：
- 本地序列/顺序
- 拥有的草稿状态
- 来源（provenance）
- 锚定到代码/上游
- 契约/不变量
- 维护策略

### 2. 草稿补丁（或微提交）

代理产生的每个有意义的编辑都应当被捕获为一个拥有的、可重放的单元。属性：
- 与一个车道关联
- 可归因于一个代理/模型/会话
- 基于特定修订版本
- 可还原和可重放
- 后续可安全压缩

这避免了共享脏工作树所有权不明确的模型。

### 3. 爆发（Burst）

代理在追求一个子任务时，通常会发出几个快速的、连贯的编辑。一次爆发将这些时间上相邻的草稿补丁分组为一个操作工作片段。

### 4. 发布提交

面向人类的提交可以从一个或多个草稿补丁或爆发中压缩而来。这提供了一个两层模型：
- 用于并发和维护的操作历史
- 用于人工审查和共享的发布历史

### 5. 维护包

每个本地增量都应携带一个丰富的上下文包，包含：
- 变更的意图
- 与上游的关系
- 测试覆盖范围
- 依赖关系
- 替代方案考虑
- 维护历史
