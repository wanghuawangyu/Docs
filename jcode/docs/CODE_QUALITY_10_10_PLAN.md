# 代码质量 10/10 计划

本文档定义了 jcode 的质量目标、达到该目标所需的标准，以及在不破坏产品稳定性的情况下达到目标的分阶段执行计划。

## 目标

将 jcode 从当前约 7/10 的整体代码质量提升到持续的 9+/10 工程标准，在日常开发中达到感觉像“10/10”的实用目标：
- 干净的构建
- 清晰的模块所有权
- 小而可维护的文件
- 低风险的重构
- 强大的测试
- 压力下可预测的行为
- 防止回归的严格 CI 护栏

因为 jcode 是一个快速移动的产品，“10/10”并不意味着“完美”。它意味着：
- 缺陷比引入更容易预防
- 贡献者能快速理解代码归属
- 代码库能抵抗架构漂移
- 风险区域有良好的测试和可观测性
- 质量不依赖于记忆或英雄主义

## 当前问题

今天代码库中观察到的主要问题：

### 1. 超大模块

几个文件对于长期可维护性来说太大了。当前主要热点包括：
- `src/provider/openai.rs`
- `src/provider/mod.rs`
- `src/agent.rs`
- `src/server.rs`
- `src/tui/ui.rs`
- `src/tui/info_widget.rs`
- `tests/e2e/main.rs`

这些文件一次性做了太多事情，造成了审查、测试和 onboarding 的摩擦。

### 2. 警告和死代码债务

代码库当前容忍显著的警告预算，而不是以无警告构建为目标。还存在多个宽泛的 `allow(dead_code)` 抑制，隐藏了漂移。

### 3. 失败路径上的一致性不足

代码库包含许多 `unwrap`、`expect`、`panic!`、`todo!` 和 `unimplemented!` 的使用。有些在测试中是有效的，但生产代码应该更加防御性和明确。

### 4. 测试集中

有很多测试，这很好，但一些测试覆盖率集中在非常大的文件中，尚未提供理想的故障隔离。

### 5. 护栏存在但还不够严格

代码库中已经有有用的质量基础设施，但应该收紧，以便质量随时间自动改善。

## “10/10”完成定义

当代码库达到以下状态时，我们认为此计划成功：

### 构建和 lint 质量
- `cargo check --all-targets --all-features` 干净通过
- `cargo clippy --all-targets --all-features -- -D warnings` 干净通过或非常接近，仅有少量有正当理由的例外
- `cargo fmt --all -- --check` 通过
- 警告计数接近零并积极向下收紧

### 结构质量
- 没有生产文件超过 1200 LOC 而无记录原因
- 大多数生产文件低于 800 LOC
- 大多数函数保持在 100 LOC 以下，除非复杂性有明显理由
- 主要领域有清晰的边界和所有权

### 可靠性质量
- E2E 测试按功能拆分，而不是集中在巨型文件中
- 关键状态转换有针对性测试
- 重载、流式、工具执行和 swarm 协调有明确的故障模式覆盖
- 存在针对内存、socket 生命周期和重连/重载行为的长期可靠性检查

### 安全质量
- 生产代码中的 `unwrap`/`expect` 数量显著减少
- 错误上下文丰富且可操作
- 敏感操作有审计跟踪
